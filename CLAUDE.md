# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Context

Academic final-year project entering [BirdCLEF+ 2026](https://www.kaggle.com/competitions/birdclef-2026) (Kaggle code competition, deadline 2026-06-03). Task: identify 234 species (162 birds + 35 amphibians + 28 insects + 8 mammals + 1 reptile) from 5-second audio segments of Pantanal wetland field recordings. Metric: macro-averaged ROC-AUC, skipping classes with no positives.

The team runs **two parallel approaches**; this repo holds **Route B**:

- **Route B (this repo)** — frozen pretrained Google Perch v2 encoder + trainable MLP head (transfer learning / linear probing methodology).
- **Route A (teammate, not in this repo)** — EfficientNet-B2 + self-training loop with MixUp and pseudo-labels.

Final submission must run on **Kaggle CPU notebook, 90 min, no internet**.

## Pipeline (Phase 1 done, Phase 2 done, Phase 3 pending)

The pipeline is deliberately split because the encoder runs in a different environment than the head. Phase 3 (submission notebook) is the only remaining work — see "What's Not Here Yet" at the bottom.

**Phase 1 — Embedding extraction (Kaggle GPU, ~47 min, runs once)**
- Notebook: `new/kaggle_01_extract_embeddings.ipynb` (uploaded to Kaggle, not run locally)
- First cell *must* `!pip install -q -U "tensorflow==2.20.0" perch-hoplite`, then **manually restart the kernel** before running the rest. The notebook's own assert (`tf.__version__.startswith("2.20")`) catches this.
- Loads `perch_v2_gpu` via `perch_hoplite.zoo.model_configs.load_model_by_name()`
- Frames audio to 5-sec windows, batches 32 frames at fixed shape (avoids XLA recompilation), runs `model.batch_embed`
- Output: 5 files in `embedding/` (see Data Contracts below)
- Already committed once as a public Kaggle kernel; the cached output Dataset can be re-pulled via `kaggle kernels output ejoshfu/google-bird -p ./embedding/` if `embedding/` is wiped.

**Phase 2 — Head training (local RTX 5070, ~2 min total for 3 variants)**
- Script: `new/train_mlp_local.py` runs ablation over 3 head variants:
  - `A_linear_bce`: `nn.Linear(1536, 234)` + BCEWithLogitsLoss (~360k params)
  - `B_mlp_bce`: 3-layer MLP with BatchNorm + Dropout(0.3) + BCE (~1.57M params)
  - `C_mlp_focal`: same MLP, focal BCE loss (γ=2.0, α=0.25)
- Script: `new/eval_domain_auc.py` re-trains the same 3 variants but additionally tracks `ta_auc` (train_audio val portion) and `sc_auc` (soundscape val portion) per epoch — this format mirrors Route A's reporting style.

## Data Contracts (`embedding/`)

These five files are produced by Phase 1 and consumed by Phase 2. The shapes are load-bearing — every script asserts them:

| File | Shape / dtype | Meaning |
|------|---------------|---------|
| `train_audio_embeddings.npy` | `(233101, 1536)` float32 | Perch v2 embeddings for every 5-sec frame across 35,549 train_audio files |
| `train_audio_labels.npy` | `(233101, 234)` float32 | multi-hot labels (primary + secondary). All frames of one file share the same label |
| `train_audio_meta.parquet` | columns `filename`, `frame_idx` | Maps each row back to source file. Used as `groups` in `GroupShuffleSplit` to prevent train/val leakage |
| `soundscape_embeddings.npy` | `(1478, 1536)` float32 | One embedding per labeled 5-sec segment from `train_soundscapes_labels.csv` |
| `soundscape_labels.npy` | `(1478, 234)` float32 | Multi-hot from semicolon-separated labels |

In Phase 2 scripts, train_audio + soundscape are concatenated (`X_all` shape `(234579, 1536)`) and split via `GroupShuffleSplit(test_size=0.15, random_state=42)`. Soundscape rows use `f"__ss_{i}__"` as their group, so they cannot collide with train_audio filename groups.

## Critical Environment Constraints

These are well-tested workarounds to non-obvious problems. Don't undo them without reason:

1. **Perch v2 cannot run locally on Windows.** TensorFlow dropped native Windows GPU after 2.10, but Perch v2 requires TF ≥ 2.20 (the SavedModel embeds StableHLO v1.11 bytecode). Local Phase 1 is therefore impossible — that's why Phase 1 is fixed to Kaggle.

2. **RTX 5070 is Blackwell (sm_120).** TF 2.20's CUDA binaries don't include sm_120 PTX, and `tf-nightly` import is broken at the time of writing. PyTorch 2.11 stable + CUDA 12.8 (`pip install torch --index-url https://download.pytorch.org/whl/cu128`) is the only working configuration for local GPU work. Do not install TF in `.venv-torch`.

3. **Perch has two variants with the same weights but different XLA targets.** Use `perch_v2_gpu` on the Kaggle GPU notebook, `perch_v2_cpu` on the Kaggle submission notebook. Mismatching variant and platform raises `NotFoundError: The current platform CUDA is not among the platforms required by the module: [CPU]`.

4. **`train_soundscapes_labels.csv` has time strings, not floats.** `start` / `end` columns arrive as `"00:00:05"` strings; multiplying by sample rate triggers Python string repetition, not numeric multiplication. Always convert with `pd.to_timedelta(col).dt.total_seconds()` after `pd.read_csv`.

5. **XLA recompiles per batch shape.** When iterating audio files individually, batch sizes vary (some files have 2 frames, others 20), so XLA recompiles for each new size at ~3-5s each. Phase 1 buffers frames to a fixed `BATCH_SIZE=32` before calling `batch_embed` — this gave a ~10× speedup over the naive per-file loop.

6. **Caches must stay inside the project directory.** Per durable user preference, do not let `kagglehub`, `huggingface_hub`, `pooch`, `tfhub`, or pip pollute `%USERPROFILE%\.cache`. When applicable, set `KAGGLEHUB_CACHE`, `HF_HOME`, `TFHUB_CACHE_DIR`, `XDG_CACHE_HOME`, `PIP_CACHE_DIR` to `c:\birdCLEF\cache\<tool>` before any such tool is imported.

## Conventions

- All scripts hardcode `SEED=42` and use `GroupShuffleSplit(test_size=0.15, random_state=42)` so results are reproducible across runs and machines.
- Validation metric is **macro ROC-AUC, skipping classes with no positives in val** — this matches the competition's evaluation. Never use plain accuracy; multi-label accuracy is meaningless here.
- Outputs always land in `c:\birdCLEF\new\`. Scripts use raw Windows paths (`Path(r"C:\birdCLEF\...")`); don't introduce relative paths or POSIX paths in script code.
- `train_run.log`, `*.json`, and `*.png` in `new/` are generated artifacts — they are valid for the current best results and should not be hand-edited.
- The user is an ML beginner; favor running things end-to-end first, explaining mechanics afterwards. Notebook-style or single-file scripts are preferred over deep package abstractions.

## Experiment Log Convention (MANDATORY)

**This is a hard rule.** Every optimization attempt, research detour, and architectural decision must be recorded in `new/EXPERIMENT_LOG.md` **as it happens**, not after the fact. Claude must update the log **during** work — write the entry header and motivation **before** starting coding, record dead ends and detours **as they occur**, and add results **immediately** after each run. Never batch-write the log at the end of a conversation.

### Real-time logging cadence

- **Before starting**: create the EXP entry with date, ID, and motivation
- **During work**: append dead ends / 彎路 the moment they happen (e.g., "tried X, got error Y, root cause Z")
- **After each run**: append results table immediately
- **Before ending conversation**: ensure the entry has all 9 sections filled in

### What to record (each entry)

1. **Date + experiment ID**: e.g., `### 2026-05-30 EXP-004: ProtoSSM chain research`
2. **Motivation**: why this experiment was attempted (Kaggle discussion insight, paper, user request, etc.)
3. **What was tried**: code changes, hyperparameters, architecture decisions
4. **Dead ends / 彎路**: things that looked promising but didn't work, with the reason. This is the most valuable part — prevents repeating mistakes across conversations.
5. **Results**: local val AUC, LB score (if submitted), training time, comparison to baseline
6. **Files created/modified**: full list of scripts, weights, configs touched
7. **File backups**: before overwriting any model weight or result file, note the previous file's identity (e.g., "`mlp_best.pt` was from B_mlp_bce, val AUC 0.9646"). If a script is substantially rewritten, note the before-state.
8. **Discussion / 討論**: key insights, references to papers/Kaggle posts, reasoning behind decisions
9. **Next steps**: what to try next based on this experiment's outcome

### Environment isolation rules

- All new venvs, caches, pip installs, or temp files stay under `c:\birdCLEF\`. Never pollute `%USERPROFILE%`.
- Never modify `.venv-torch` package versions or `embedding/` data without explicit user approval and backup.
- New experiment scripts get their own output filenames (e.g., `proto_ssm_results.json`, not overwriting `ablation_results.json`).
- If an experiment requires a new Python package, install it in `.venv-torch` and record the package + version in the log entry.
- **External data** (Xeno-canto, AnuraSet, iNaturalist, etc.) goes into `c:\birdCLEF\external\{source}\` — never mix with competition data in `birdclef-2026/`.
- **External embeddings** go into `c:\birdCLEF\embedding_external\` — never mix with Phase 1 outputs in `embedding/`.
- Before overwriting **any** `.pt` weight file or `.json` result file, record in the log: (a) the file's previous identity (which experiment produced it), (b) its key metric. If in doubt, rename the old file with a `_bak` suffix first.

### Why this matters

This project spans multiple Claude conversations. Each conversation starts with no memory of previous sessions except what's in these files. The experiment log is the **single source of truth** for what was tried, what worked, and what failed. Without it, future sessions will repeat dead ends or overwrite good results.

This rule ensures every conversation can pick up where the last one left off, even if context is lost.
